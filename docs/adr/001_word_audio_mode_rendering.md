# ADR 001: Word Audio Modeの描画ロジックに関する設計判断

## ステータス
**承認済み (Accepted)**

## コンテキスト
Word Audio Mode（英単語聞き流し動画）の動画生成プロセスにおいて、過去に「コードの継ぎ足しにより描画が重なるバグ（二重表示）」が発生していた。このバグは、背景画像の上にテキストを複数回重ねて描画する処理が、各フレームの生成時に正しく初期化されていなかったことに起因していたと考えられる。

現在、`video_gen.py` の `generate_word_audio_video` 関数内で描画ロジックがシンプルに再構築されている。この再構築されたロジックが、期待される「シンプルかつ洗練されたスタイル（動画1）」を維持できているか、また二重描画のバグが完全に解消されているかを確認する必要があった。

## 決定事項
現在の `video_gen.py` における `make_frame` 関数の描画ロジックを維持し、追加の修正は行わない。

## 理由
コード解析の結果、以下の点が確認されたため、現在のロジックは安全かつ要件を満たしていると判断した。

1. **フレームごとの完全な初期化**: 
   `make_frame` 関数内で、毎フレーム必ず `img = Image.new("RGB", (1280, 720), (0, 0, 0))` を呼び出して真っ黒な新規キャンバスを作成している。これにより、前のフレームの描画結果が残って二重描画されるバグは原理的に発生しない。
2. **背景画像の動的生成**: 
   以前は `bg_img.copy()` を使ってベース画像を使い回す設計があったが、現在は `Image.new` で完全にゼロから生成した上で、ロゴやテキストを描画する方式になっている。
3. **描画処理のシンプルさ**: 
   単語（英語）、意味（日本語）、ID（No.xxxx）の描画座標とフォント指定が明確に分離されており、無駄な重ね書き処理が存在しない。

## 結果
- **バグの解消**: 二重描画バグは完全に解消されていることがコードレベルで証明された。
- **パフォーマンスへの影響**: 毎フレーム `Image.new` を呼び出すことによる若干のオーバーヘッドはあるが、MoviePyの処理全体から見れば微々たるものであり、動画の品質と安定性を優先する現在の設計は妥当である。
- **今後のアクション**: 長時間動画（30分〜1時間）の生成時にメモリリークが発生しないか、実際の運用プロセスで監視を続ける。
